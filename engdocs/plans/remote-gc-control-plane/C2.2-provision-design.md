## 0. Error-rendering contract (read first)

Every fatal `stderr` message inside `doRigAddWithResult` today has the shape
`"gc rig add: " + <detail> + "\n"` (verified against all ~30 fatal sites, :230-:625).
Post-extraction, **Provision returns `error` whose `Error()` is exactly `<detail>`**,
and the cmd/gc wrapper prints it at ONE site:

```go
fmt.Fprintf(stderr, "gc rig add: %v\n", err)
```

The one exception is the config-load failure (`:260`,
`"gc rig add: loading config: %v\n"`), which the wrapper prints itself because the
config is loaded *by the caller* (Deps.Cfg is a required field, `deps.go:27`).

Non-fatal output rides two channels:

- **stdout banner lines** → `OnStep(ProvisionStep{Name, Detail})`, CLI renders
  `fmt.Fprintln(stdout, step.Detail)` — identical bytes to today's
  `w := func(s string) { fmt.Fprintln(stdout, s) }` (`:468`).
- **stderr warn lines emitted mid-provision** → `OnStep(ProvisionStep{..., Warn: true})`
  with `Detail` = the text after `"gc rig add: "`; CLI renders
  `fmt.Fprintf(stderr, "gc rig add: %s\n", step.Detail)`. All five in-Provision warn
  sites (`:472,:477,:479,:483,:488`) start `"gc rig add: warning: ..."`, so this holds.
- **stderr warn lines in the post-write tail** (`:590-627`) → stay verbatim inside the
  CLI's `Deps.PostProvision` closure (see §2), so no mapping is needed at all.

Warn steps are also appended to `ProvisionResult.Warnings` (Detail strings) for the
API consumer; the CLI ignores `Warnings` and renders only via `OnStep`.

## 1. Provision signature & body decomposition

```go
func Provision(deps Deps, req ProvisionRequest) (config.Rig, ProvisionResult, error)
```

(unchanged from the C2.0 scaffold, `internal/rig/deps.go:103`). `req` carries the RAW
CLI inputs: `Prefix` and `DefaultBranch` un-lowercased/un-trimmed, `Includes`
un-trimmed — Provision performs the normalization internally so warning text that
echoes raw flags (`:483 "--prefix=%s ignored"`, `:488 "--default-branch=%s ignored"`,
`:477/:479 "--include flags %v ignored"`) stays byte-identical. `req.Name` is the
already-derived name (wrapper does `filepath.Base(rigPath)` when the flag is empty,
mirroring `:244-247`) because `validateRequest` requires it (`deps.go:117`).

func doRigAddWithResult(fs fsys.FS, cityPath, rigPath string, includes []string,

## 5. Behavior-preservation RISKS

- **R1 — config-load precedence inversion (accepted, flagged).** Today the `req.Path`
  stat (`:227`) precedes the config load (`:258`); with `Deps.Cfg` required, the
  wrapper loads config first. Sole observable divergence: rigPath invalid AND
  city.toml broken → the emitted error flips from the stat error to
  `loading config: ...`. No test covers the double-failure; every single-failure case
  is unchanged. Mitigation if byte-purism is demanded: replace `Cfg *config.City`
  with `LoadCfg func() (*config.City, error)` and load at today's exact point — not
  recommended (complicates every caller for an untested corner).
- **R2 — dolt-config registration window widens.** Register moves from after
  ComposePacks (`:277`) to before Provision; on a ComposePacks failure the registry
  is now registered-then-cleared instead of never-registered. `cityDoltConfigs` is an
  in-process `sync.Map` read only by the beads-init path and cleared by the wrapper's
  `defer` on every exit; no goroutine observes the window in CLI or tests. The pairing
  stays in one lexical scope (plan invariant 3). Do NOT move the pair inside Provision
  — that would force injecting both funcs and re-splitting them around ComposePacks.
- **R3 — the deferred `commit()` closure.** Two subtleties: (a) call #2 (`:372`)
  REASSIGNS `commitRigImports` — Provision must overwrite, not accumulate (safe: call
  #2 is reachable only when call #1 returned nil commit, because non-empty
  `explicitRigImports` short-circuits the default branch at `:360`); (b) the commit
  must run strictly between the config write and `WriteRoutes`, inside the
  rollback-guarded region, with the step-14 snapshot already covering `packs.lock`
  (`:887-891`). Getting this order wrong silently breaks plan invariant 1.
- **R4 — `StoreContract`/`DoltSkip` must be funcs, not booleans.** `initDirIfReady`
  can seed provider/scope state mid-flow (`seedDeferredManagedBeadsErr`,
  `initDirIfReadyEnsureBeadsProvider`, `beads_provider_lifecycle.go:247-310`), and the
  flow re-evaluates `cityUsesBdStoreContract` after init (`:532`). Hoisting to a bool
  at Deps-build time risks a stale value on exactly the managed-dolt paths the
  deferred messaging exists for.
- **R5 — InitAndHook error swallowing.** In the deferred fallback (`:534`), a failing
  `initAndHookDir` is deliberately reported as `Beads init deferred to controller`,
  not an error. A "clean" implementer instinct to propagate it breaks
  `TestDoRigAddSqliteCityCreatesBdBackedRig`-class tests.
- **R6 — ValidateRigs placement.** It must stay between nextCfg construction and
  `MkdirAll` (`:385-397`): validation failures must leave the filesystem untouched
  (`TestDoRigAdd_CandidateValidationErrorDoesNotCreateMissingRig`,
  `cmd_rig_test.go:922`), and it must stay gated on
  `needsValidation = !reAdd || reAddNeedsConfigWrite` — a plain re-add skips it.
- **R7 — adopt vs re-add branch coupling.** Re-add uses the existing prefix and
  ignores `--prefix` (`:310`, warning `:483`); adopt validates store files (`:399`)
  and gets its own prefix-mismatch message (`:423`); the fresh-add `.beads` guard
  (`:444`) is gated `!reAdd && !adopt`. These three booleans interleave across steps
  6-13 — extract the flow as one function, do not split per-branch helpers in C2.2.
- **R8 — import-cycle traps.** Anything referencing `package main` symbols must be
  injected, never moved: `ensureBundledRigImportsInstalled` (go:embed +
  `collectAllImportsFS` + `syncImports`-var test seams), `initDirIfReady`/
  `initAndHookDir`, `normalizeCanonicalBdScopeFiles`, `prepareRigAdoptProviderState`,
  `writeAllRoutes` (hardcoded `fsys.OSFS{}`/`os.*`, `rig_beads.go:63`),
  `writeBeadsEnvGTRoot` (via `ensureBeadsDir`), `installBeadHooks`,
  `ResolveFormulas`, `rigReloadControllerConfig`/`rigWaitForStoreAccessible`.
  Conversely `internal/rig` may import `internal/{config,packman,builtinpacks,git,
  fsys,beads/contract,hooks}` freely.
- **R9 — test-seam vars.** Deps closures must reference cmd/gc package vars at call
  time (`writeAllRigRoutes`, `rigReloadControllerConfig`, `rigWaitForStoreAccessible`,
  and transitively `syncImports`/`installLockedImports`/`writeImportLockfile` inside
  `ensureBundledRigImportsInstalled`) so var-override tests keep working; and the
  moved pure helpers need cmd/gc aliases (§2) because six of them are called directly
  from cmd/gc test files.
- **R10 — final-result shape.** The returned `config.Rig` comes from the post-write
  `cfg.Rigs` lookup (`:637-641`), so a fresh add returns `Prefix` as the *stored*
  (possibly empty) prefix, not the effective one, and the unreachable-in-practice
  fallback (`:642-648`) sets `Suspended` where the constructed rig set
  `SuspendedOnStart`. `rigAddJSONSummary` and `TestRigAddJSONEmitsOnlySummary` render
  from this struct — reproduce it verbatim.

## 6. Does `ComposePacks`'s deferred `commit()` fit the Deps shape? — YES

Scaffolded signature (`deps.go:38`):

```go
ComposePacks func(cityPath string, imports []config.BoundImport) (pinned []config.BoundImport, commit func() error, err error)
```

`ensureBundledRigImportsInstalled` (`cmd_rig.go:726`):

```go
func ensureBundledRigImportsInstalled(cityPath string, imports []config.BoundImport) ([]config.BoundImport, func() error, error)
```

These are **identical** — cmd/gc wires the function value with zero adaptation. The
closure-return shape is exactly what the ordering invariant needs: resolution
(lockfile sync + repo-cache hydration, `:741-750`) happens eagerly so errors surface
before mutation; the returned `commit` (which does `writeImportLockfile` +
`installLockedImports`, `:752-760`) is held by Provision and invoked at step 16, after
the config write, under the step-14 snapshot (which covers `packs.lock`). `commit` may
be nil (no bundled imports, `:738-739`); Provision nil-checks as today (`:575`). The
call-#2 reassignment (R3a) is plain local-variable semantics inside Provision. **No
signature change needed.** The only scaffold deltas required for C2.2 are the four new
Deps fields and the `ProvisionContext.Cfg` field listed in §1.
