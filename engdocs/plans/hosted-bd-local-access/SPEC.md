## Problem

A hosted Beads project is a per-tenant database reachable only through the beads-gateway, which authenticates a short-lived identity token carried inside the database wire handshake (MySQL username slot / pg password slot) and then byte-splices to the backend; bd already has the fail-closed credential machinery and the gateway already enforces the full tenancy join — yet no local user can connect. Four connectivity gaps compose into total failure: (1) the gateway presents a SPIFFE-only certificate that bd's binary TLS knob can structurally never verify; (2) the endpoint is tailnet-only with no public name; (3) the documented "Connection panel" recipe surface does not exist in shipped beads-web and the one-time `bd init --server` step is broken against gateways; (4) all NEW projects since Jul-13 land on Postgres backends that the deployed MySQL-wire-only gateway cannot serve externally — and bd's native pg backend has no hosted mode (it runs DDL on every open). Adversarial review exposed two further gaps that any naive connectivity fix would weaponize: EIAs carry no destination binding, so repo-local connection config is a credential-exfiltration channel (a cloned repo can direct the victim's own helper to mint a real token and hand it to an attacker's host), and the CLI's at-rest custody story is false (session token, DPoP private key, and cached EIAs live in one 0600 file — the file alone mints). This spec therefore defines three contracts, not one: the wire/TLS contract (direct bd→gateway over a publicly-trusted cert, per engine), the destination-trust contract (a user-level trusted-gateway allowlist that repo config can never override, plus destination-aware minting), and the credential-custody contract (OS-keyring key separation, a server-side token-lifetime ceiling, and membership-revocation severance) — with no local proxy, no edge TLS termination, no dialect translation, and no static secrets.

## Goals

- A user on any laptop reaches ONE hosted project with three commands — `gasworks login`, `bd attach <copy-pasted recipe>`, `bd list` — in under 3 minutes, with zero pasted env vars on the human golden path, zero static secrets, and zero TLS artifacts (no CA bundles, no fingerprints, no flags).
- Identical contract for Dolt-engine and Postgres-engine projects: same login, same attach command shape, same panel; only the wire codec, credential slot, and schema-ownership mode differ, chosen by tooling from the project's backend_id engine.
- Every byte of hosted access flows through the beads-gateway choke point (org→account→project tenancy join, workspace gate, RTBF fence, spend-gate read-only downgrade, credential containment); direct-to-backend from user machines is impossible by construction because clients never learn backend coordinates.
- Destination trust: bd never transmits a minted credential to a gateway host outside a user-level trusted-gateway allowlist, and repo-local configuration can never select the mint destination — a hostile cloned repo hard-fails before the credential command runs.
- Credential custody at rest: by GA the DPoP private key lives in an OS keyring separate from the session token, plaintext EIAs are not cached at rest, and every mint is audited fail-closed; token lifetime stays 90s and is additionally capped server-side so even a forged-exp token is bounded.
- Least privilege by GA: recipes mint project-pinned scopes (beads:read#prj_<id> / beads:write#prj_<id>) so a token grants exactly one project, not the whole org; org-membership removal severs mint capability within one token exchange, not one session lifetime.
- Machine/CI path is character-identical to the human path: an API key (GASWORKS_API_KEY) makes the same credential command mint via /sts/v0/machine, and non-interactive trust (BD_TRUST_WORKSPACE=1 / --accept) replaces the interactive attach ceremony; no recipe divergence.
- Long-lived clients (gc controller, daemons, editors) work correctly: fresh token minted per physical dial with jittered refresh and backoff, auth-failure retry re-mints, and (tuning slice) splice lifetime decoupled from token expiry under a periodic authz revalidation loop.
- The one-time workspace materialization becomes a designed, idempotent, self-diagnosing command (`bd attach`) with an explicit trust ceremony, a defined committed-metadata/second-user/CI story, and server-authoritative project identity — instead of an undocumented broken init incantation.
- Every failure names its layer and exactly one remedial command (diagnostic contract + `bd doctor`), including the failure modes actually observed live: pre-auth 'TLS required', mint-side entitlement 403, and wrong-org 1045.

## Non-goals

- No SQL dialect translation anywhere: the gateway remains an auth-then-transparent-byte-splice (gateway.go:1-11); cross-engine connects are refused with a redirect message, never translated.
- No local proxy daemon (pscale-connect / cloud-sql-proxy pattern): it adds a resident process, a plaintext loopback hop, and a new failure class, solely to solve TLS trust and cred injection — both solved natively here.
- No edge TLS termination: forbidden by the gateway's own fail-closed design (the credential rides the handshake; guardAddr refuses non-loopback listeners without local TLS; S4a §5 'LB-terminated TLS does NOT satisfy it').
- No direct-to-RDS/backend recipes for pg projects, ever — even as a stopgap: it leaks backend coordinates, bypasses the entire enforcement stack, and demotes per-project DB users to the sole tenant boundary.
- No skip-verify or otherwise-unverified TLS mode in bd, not even behind a flag: the credential is inside the handshake, so an active MITM harvests a replayable token AND the full ledger stream. The deferred enterprise CA-pin path is NOT an exception — it must fully verify the chain against the pinned CA and require exact SPIFFE URI-SAN equality, with adversarial tests as a merge gate.
- No lengthening of the EIA TTL as a performance lever: mint load is engineered with caches, jittered refresh, and per-dial reuse; the wire token stays at 90s for GA, and any future raise is a separately-gated security decision bounded by the gateway's server-side exp-iat ceiling.
- No new durable token class: bts_ static tokens are bounded now (mandatory expiry, default-off for new projects) and sunset, not extended; no 'connection password' invention.
- No reliance on recipe secrecy: recipes carry zero credential material, but they ARE targeting data (org + project + gateway). Safety comes from the destination allowlist and mint ceremony, not from hiding recipes — the spec never claims recipes are 'safe to commit' as a security property.
- No offline / degraded-network operation; hosted access requires a live STS.
- No private-CA distribution in the hosted recipe: WebPKI on the gateway supersedes the documented earlier intent; a custom-CA + SPIFFE-pin bd knob ships later for private/enterprise deployments only.
- No changes to embedded/local (sqlite/embedded-dolt) bd workflows; this spec touches only server/gateway mode.

## Architecture


### Summary

Spine unchanged from the draft: bd dials the gateway DIRECTLY over one public hostname whose cert is publicly trusted, using the credential machinery bd already has — no local proxy (a resident daemon + plaintext loopback hop added purely to launder TLS trust and cred injection, both fixed at their roots here), no edge TLS termination (forbidden because the credential rides the handshake), no private-CA distribution to laptops (WebPKI solves it with zero client code). The adversarial review forced three structural corrections that this final spec adopts as first-class contracts rather than mitigations. (1) DESTINATION TRUST: because EIAs are destination-unbound (aud=product only) and connection config was to live in the repo, the draft's 'recipe is safe to commit' posture handed attackers a mint-and-exfiltrate channel; the fix moves the gateway HOST out of repo-trusted config entirely — bd resolves it from a user-level trusted-gateway allowlist (default entry shipped in the release), plain bd commands hard-fail (no prompt) on unknown hosts, trust is established only via the explicit `bd attach` ceremony, and the helper itself becomes destination-aware (gasworks refuses to mint for unlisted hosts). (2) CREDENTIAL CUSTODY: the DPoP file-theft mitigation was false (session token + DPoP private key + plaintext EIA cache co-located), so keyring separation + no-plaintext-EIA-at-rest becomes a GA-blocking slice; the EIA TTL stays 90s (the draft's 15m raise is dropped — revalidation bounds principal revocation, NOT token theft, so TTL is the only replay bound and mint load is engineered with jitter/caches instead); the gateway gains a server-side exp-iat ceiling so even a forged-exp token from a signer compromise is lifetime-bounded; and membership removal severs human mint within one exchange (auto bulk-revoke + exchange-time recheck). (3) HONEST STAGING: the S1 cert swap becomes a client-inventoried dual-cert migration with a single secret owner and hot-reload (the 'zero gateway code change' claim is retired); pg projects get a real hosted client mode in bd (today's pg backend DDLs on every open — unusable against a gateway) plus a schema-field wire contract, and the pg slice is re-cut to tailnet-first depending only on the client and least-privilege slices, so the majority (new-project) cohort is not gated behind public exposure.

### Connection model

Direct bd → gateway TLS, end to end; strictly no local proxy, no edge TLS termination. One hostname — gw.beads.gascity.com — two ports: :3306 (MySQL wire, Dolt-engine projects; in-pod :3307) and :5432 (pg wire, Postgres-engine projects; in-pod -pg-addr). Path: DNS A-record → haproxy in pure L4 tcp passthrough (today's corp-public-ha pattern) → loopback-only NodePort → gateway pod, which keeps terminating TLS itself (guardAddr fail-closed requirement). Rollout is tailnet-first: S1 adds a split-DNS/tailnet A-record for gw.beads.gascity.com so the WebPKI cert (mintable via ACME DNS-01 without public reachability) verifies for on-tailnet design partners; S7 adds the public A-record + public haproxy bind. 1 inbound conn = 1 backend conn (transparent splice, no gateway pooling in v1); bd's own pool becomes effective once S8 stops killing splices at token expiry. Hard prerequisites for the PUBLIC bind: (a) PROXY protocol v2 ingestion in the gateway, accepted ONLY from an allowlist of trusted upstreams (the haproxy loopback NodePort source) — PROXY headers from any other peer are rejected/ignored and the real TCP source used, with a test that a direct client cannot spoof its attributed IP; (b) retune/retire the haproxy per-source conn-cur 64 stick-table once the gateway attributes real client IPs; (c) O(1) connect-time authz (below); (d) all existing pre-auth hardening retained (8KiB handshake clamp sized for multi-KB JWTs, 10s handshake deadline, 4096 conns/pod, 128/IP, 512 accepts/s, scrubbed access-denied errors, per-connection panic containment).

### Credential flow

HUMAN: (1) `gasworks login` — browser OIDC with --device-auth fallback, explicit org selection frozen into an 8h DPoP-bound gcs_user session. Custody (S4, GA gate): the DPoP private key moves to the OS keyring (Keychain/DPAPI/kernel-keyring), SEPARATE from the session token; plaintext EIAs are no longer cached at rest; where no keyring exists the file is encrypted with an OS-protected key and the threat model says so honestly. (2) The recipe's credential command is `gasworks getToken beads --org <org> --project prj_<id> --gateway gw.beads.gascity.com`; per invocation it performs the RFC 8693 exchange with a DPoP proof, using per-(org|product|scope) caching with JITTERED early refresh (re-mints spread across the TTL window, never on the boundary), exponential backoff on STS 5xx/429, and serve-last-good-until-true-expiry through brief STS outages. The EIA is a 90s RS256 JWT, aud=beads, org server-stamped, signed by eia-signing-beads; the gateway additionally rejects any EIA whose exp-iat exceeds a hard server-side ceiling (15m) regardless of claims. (3) DESTINATION GUARD, two independent layers: bd resolves the gateway host ONLY from a user-level trusted-gateway allowlist stored outside any repo (a default entry for gw.beads.gascity.com ships in the release); repo-local metadata NEVER carries a mint destination; a plain bd command whose resolved host is untrusted hard-fails with instructions (no prompt — prompts are for the explicit attach ceremony only). Independently, gasworks getToken refuses to mint when --gateway names a host outside its own (built-in + user-managed) allowlist. EIA destination/channel binding in the token itself is tracked as future hardening. (4) bd runs the helper through the fail-closed creds ladder and presents the token as the MySQL-wire USERNAME (ApplyGatewayCredential: KindIdentity, ':@/' rejected, Gateway=true, DisableAutoStart) or the pg PASSWORD (BEADS_PG_PASSWORD_COMMAND; BEADS_PG_CREDENTIAL_COMMAND stays a reserved hard error). (5) PER-DIAL: the BeforeConnect connector (ported from feat/dolt-credential-command) stamps a fresh cached token on every new physical connection; a 1045 auth error invalidates the cache so the retry re-mints — this makes long-lived clients correct without a proxy; the pgx path gets the symmetric fresh-password-per-dial + auth-failure-invalidate. MACHINE/CI: GASWORKS_API_KEY=beads_live_... makes the identical command exchange via /sts/v0/machine; workspace-bound SP keys carry workspace:<id> resource refs; /sts/v0/delegate covers sub-agents; BD_TRUST_WORKSPACE=1 (honored only when GASWORKS_API_KEY is present) or `bd attach --accept` provides non-interactive trust. LEAST PRIVILEGE (S3, GA gate): pin-aware STS intersection (grant base#prj_<id> iff bare base ∈ ceiling — gateway enforcement already exists), recipes default to pinned read+write, --read-only mints read#pin only. MEMBERSHIP SEVERANCE (S3, GA gate): org-membership-change events wire to automatic session bulk-revoke, AND human sessions re-check membership at every /sts/v0/token exchange (as service sessions already re-check SP liveness) — removal severs mint within one exchange, not one 8h session. PERSISTENCE: bd attach writes credential_helper="gasworks" + org/project params (allowlisted binary, no shell, exact argv, strict charset) into metadata.json — but NOT the host; trust-on-first-use is recorded OUTSIDE the repo keyed by workspace path + hash(helper, args, gateway-ref, database), invalidated on any change. Freeform BEADS_DOLT_CREDENTIAL_COMMAND stays env-only and always wins. REVOCATION LADDER: EIAs unrevocable by design, bounded by 90s TTL + server ceiling; sessions hash-stored, DPoP-bound, cascade-revocable with fail-closed root-walk; SP disable propagates within one TTL; membership removal within one exchange (S3); bts_ never emitted, expiry-bounded now, sunset in S9; every mint audited synchronously fail-closed before signing.

### TLS trust model

Resolve the impasse SERVER-side; bd's strict verification is the asset we keep, and bd works with ZERO TLS code changes on the hosted path: BEADS_DOLT_SERVER_TLS=true → system-root + hostname verification against the DNS SAN; pg leg sslmode=verify-full identically. DECISION (supersedes the documented private-CA intent): the gateway presents a PUBLICLY-TRUSTED WebPKI certificate for gw.beads.gascity.com via ACME DNS-01 (cert exists before any public exposure). The swap is a MIGRATION, not a leaf overwrite — the draft's 'zero gateway code change' claim is retired: (a) CLIENT-INVENTORY GATE first: pin the running pod's image/args and enumerate every current gateway client (live hosted controllers connect today by unverified means) and exactly how each satisfies TLS, before touching the cert; (b) DUAL-CERT SNI: the gateway gains a GetCertificate callback dispatching by SNI — public name serves the WebPKI cert, in-cluster/tailnet short names keep an internal cert until every internal client is migrated to the public name — and GetCertificate doubles as hot-reload so ~60-90-day ACME renewals need no binary restart; (c) SINGLE SECRET OWNER: the gateway-tls Secret becomes cert-manager-managed (the ESO ExternalSecret for it is deleted — two controllers cannot own one Secret), with expiry alerting (30/14/7-day) and a renewal runbook replacing the out-of-band OpenBao seeding that produced the SPIFFE drift. A lapsed cert fails closed for every external client simultaneously — accepted as the correct failure mode. SPIFFE identities remain exclusively on the intra-cluster BACKEND legs (ghostunnel verify-uri), untouched. BANNED PERMANENTLY: any unverified TLS mode in bd. DEFERRED (S9, enterprise only, never in hosted recipes and structurally unselectable by panel/attach): BEADS_DOLT_SERVER_TLS_CA + BEADS_DOLT_SERVER_TLS_SPIFFE_ID via mysql.RegisterTLSConfig — and because Go's hostname verifier cannot match URI SANs this path uses a custom VerifyPeerCertificate that MUST build and verify the chain with x509.VerifyOptions{Roots: pinnedCA} (never trust rawCerts), require EXACT URI-SAN equality with the configured SPIFFE ID, reject certs with no URI SAN, and check validity — with adversarial unit tests (wrong CA, substring/extra SAN, missing SAN, expired, self-signed) as a merge gate, because a hand-written verifier bug is equivalent to the skip-verify this spec bans.

### Gateway & routing

The gateway keeps its exact shape — terminate handshake, resolve credential, authorize, dial backend with the sealed per-project credential, splice bytes — and this spec freezes its auth checklist as contract: TLS-before-credential (fail-closed guardAddr; distinguishable pre-auth 1045 'TLS required'), pre-auth codec hardening, offline JWT verify against the beads-scoped JWKS (kid eia-signing-beads-v1), beads:read floor / beads:write upgrade with per-project pin support, org→account UNIQUE pivot, project∈account, workspace-tier gate (org-default projects never stranded), RTBF fence, spend-gate read-only downgrade, custody split (Admin=nil). ROUTING: database name bd_prj_<id> (charset-validated) selects the project on the MySQL leg; on the pg leg the database param is pinned to bd_<resolved-project> and the ROUTED SCHEMA becomes an explicit wire-contract field (see backendStory). Engine mismatch is refused with a redirect message, never translated. GATEWAY CHANGES (all bounded): (1) server-side EIA lifetime ceiling — reject any token where exp-iat > 15m regardless of claims, so forged/abusive long-lived tokens are impossible and kid-removal-from-JWKS becomes a real revocation channel (plus: Transit sign-access lockdown, JWKS ConfigMap write-RBAC + change audit, alerting on anomalous mint volume/kid); (2) PROXY protocol v2 ingestion from allowlisted upstreams only, with a direct-client spoof test (public-exposure prerequisite); (3) replace AuthorizeEIA's ListProjects linear scan with one indexed GetProjectInAccount(projectID, accountID) point lookup — connect budget becomes JWT verify + 2 indexed SELECTs + AES-GCM open + one backend dial; (4) bts_ bounding: reject non-expiring bts_ tokens, per-project bts_ acceptance flag defaulted OFF for newly-provisioned projects, last-used telemetry inventory before GA; (5) deploy the merged pg leg (-pg-addr=:5432) tailnet-first (S6); (6) S8: replace SetDeadline(tc.ExpiresAt) splice-kill with idle timeout 30m + absolute cap 8h + a periodic authz revalidation loop (every 5m per live splice: session/SP liveness, account status, project membership, RTBF) — stated honestly: revalidation bounds PRINCIPAL revocation (≤5m); it does NOT bound stolen-token replay, which remains bounded only by the 90s TTL + server ceiling. Auth failures stay collapsed to one scrubbed 'access denied' on the wire; the pre-auth 'TLS required' condition remains distinguishable so bd can map it to a TLS-config remedy instead of an auth chase.

### Backend story (Dolt + Postgres)

Engine determines wire; the gateway fronts BOTH engines; translation is permanently out of scope. DOLT-ENGINE (cherry.dolt.*): bd speaks Dolt SQL over MySQL wire to gw.beads.gascity.com:3306. On the wire: TLS (WebPKI cert) → MySQL handshake with the EIA as username and bd_prj_<id> as database → authenticated splice to the project's Dolt (live k8sdolt path: ghostunnel SPIFFE-mTLS → cherry ProxySQL, routed by schema). bd config: metadata.json dolt_mode=server + database + credential_helper/org/project (host from the user-level trust store) + BEADS_DOLT_SERVER_TLS=true; Gateway=true semantics (no CREATE DATABASE probe — connect IS existence proof; no client migrations; no auto-start; ADOPT server identity, never generate). POSTGRES-ENGINE (awsusw2.pg.*, cherry.pg.*): bd uses its native Postgres backend (pgx + pgdialect) to gw.beads.gascity.com:5432 — but in a NEW HOSTED-PG MODE that this spec adds, because today's pg backend is structurally unusable against a gateway (InitSchema DDL + seeds on EVERY open; version-skew error says 'recreate the workspace'; a read-only token cannot even open the store). Hosted-pg mode (metadata backend="postgres" + hosted=true, set by attach): SKIPS InitSchema/seeds entirely (successful connect is the existence proof; the hosted provisioner owns schema DDL), performs a non-destructive version-compat READ check whose failure says 'upgrade your bd binary' (never 'recreate'), never issues DDL, and ADOPTS the server's project identity via a pg-ported identity check symmetric to verifyProjectIdentity. On the wire: TLS (sslmode=verify-full) → pg startup with database=bd_prj_<id> (server-pinned) and user as diagnostic label → AuthenticationCleartextPassword carrying the EIA in the password slot (inside TLS) → splice with gateway-pinned search_path. SCHEMA CONTRACT (new): the gateway-routed schema is an explicit field in the connection API's pg branch and the recipe; bd attach writes postgres_schema to exactly that value; a contract test asserts client schema == gateway-routed schema for bd_prj_<id>. Stated honestly: bd's own SET search_path rides the opaque splice invisible to the gateway, so on shared-pg substrates the per-project SQL user's GRANTS — not search_path pinning — are the real tenant boundary (see securityModel). PROVISIONING CONTRACT (new, S1): hosted project databases are BORN with _project_id = prj_<id> and a default issue_prefix at provision time (beads-web/provider deliverable); attach adopts them and fails with a clear provisioning-contract error if absent. STAGING (re-cut per review): the Dolt path works at S1; the pg path (S6) depends only on S2+S3 and deploys TAILNET-FIRST in parallel with the panel and public-exposure slices — new-cohort customers are not gated behind either. Until S6, the panel returns a fail-closed 'endpoint rolling out' state for pg projects; direct-to-RDS recipes are rejected permanently. Rejected alternatives recorded: force-migrating pg projects to dolt; a pg-wire translation shim; gateway-side pooling in v1 (pgbouncer-style tier is the known future component if connect concurrency approaches the 4096/pod cap).

## Security model

- **[CRITICAL] On-path interception of the credential or ledger stream via TLS downgrade or unverified connection — the credential rides INSIDE the handshake, so any unverified path leaks a replayable identity token AND full read/write session access.**
  - _Mitigation:_ No unverified TLS mode exists in bd and none will be added (permanent ban); gateway refuses non-TLS clients pre-credential (distinguishable 1045 'TLS required') and refuses to serve non-loopback without local TLS (guardAddr fail-closed); WebPKI cert verified against system roots with full hostname verification (TLSConfig="true" / sslmode=verify-full); edge is L4 passthrough only — TLS terminates on the process that reads the credential; the enterprise CA-pin path (S9) must fully verify chain + exact SPIFFE URI SAN with adversarial tests as a merge gate, so no reachable code path is equivalent to skip-verify.
- **[CRITICAL] RECIPE-DIRECTED CREDENTIAL EXFILTRATION: EIAs carry no destination binding (aud=beads only), so a hostile cloned repo whose connection config keeps the victim's real org/project but swaps the host makes the victim's own helper mint a REAL token and present it to the attacker's server, replayable against the real gateway for the full TTL. Public recipes supply the targeting data (org+project+gateway).**
  - _Mitigation (ACCEPTED 2026-07-16 — owner ruled S2-DESIGN §11 Decision 1 = single helper-contract layer and Decision 2 = `eia-helper` enforcement in scope; supersedes the earlier two-layer bd-side-allowlist / `bd attach` mitigation; authoritative detail in `S2-DESIGN.md` v4):_ The destination-trust decision moves out of bd into the **credential helper**, via a helper-enforcement contract that EVERY helper implements (both the human `gasworks getToken` and the fleet's `eia-helper`). (1) bd is a pure delegator: at the mint choke it ALWAYS injects its TRUE resolved dial host — the exact `cfg.ServerHost` it dials — into the credential command's environment (`BEADS_EXEC_INFO`, kubectl-exec-plugin style, env not argv, no loopback exemption), with a `origin:bd` marker, and guarantees **report == dial** (one `cfg.ServerHost` feeds the injected env, `buildServerDSN`, and the host-keyed credential cache), so a repo-controlled host cannot produce a mint-for-A / dial-to-B split. (2) The helper reads that host and REFUSES to mint for any host outside its allowlist, and FAILS CLOSED when an `origin:bd` mint carries no destination. A hostile repo that swaps the host makes bd report `evil.example` → the helper refuses → nothing to exfil. (3) The credential command stays env-only (never repo-persisted), so the repo controls neither the helper binary nor, via report==dial, the effective destination. (4) Recipes remain targeting data, safe to commit because of the allowlist, not secrecy. (5) EIA TTL stays 90s; **the mint-time allowlist does NOT bind the token to the destination** — a captured token still replays against the real gateway for the TTL; true destination/channel binding is deferred future hardening (Open questions). **Load-bearing caveat:** this single-layer design closes the CRITICAL only for helpers that implement the contract. A non-enforcing helper (an unpatched `eia-helper`, a wrapper/PATH-shadowed `gasworks`, or a version-skewed pre-enforce build) reopens the full vector — so the contract must be a merge-gate with adversarial tests, the fleet `eia-helper` gains enforcement as part of S2 (WP-E — ruled in scope, §11 Decision 2), and the gate ships warn-only until the exec-info-injecting bd is the default install (the fleet enforce-flip is additionally gated on WP-E being deployed). The owner accepted the two-layer→one-layer collapse on these conditions (2026-07-16).
- **[HIGH] Exfiltrated ~/.config/gasworks/credentials.json: TODAY the one 0600 file co-locates the 8h session token, the DPoP PRIVATE KEY it is pinned to, and plaintext cached EIAs — the file alone yields full mint capability for the session lifetime plus immediate replay of cached tokens. The draft's 'DPoP means the file cannot mint' assurance was FALSE.**
  - _Mitigation:_ Reclassified HIGH and made a GA-blocking slice (S4): DPoP private key moves to the OS keyring, SEPARATE from the session token; plaintext EIAs are no longer cached at rest; no-keyring platforms get file encryption under an OS-protected key with the residual documented honestly. Existing controls retained: 0600, hash-stored sessions server-side, cascade revocation with fail-closed root-walk, 8h absolute expiry, synchronous fail-closed mint audit. Until S4 lands, GA does not ship and docs make no DPoP-file claim.
- **[HIGH] Signer / JWKS as fleet-wide skeleton key: EIAs are verified OFFLINE against a local JWKS file and exp is read FROM THE TOKEN — whoever can invoke the OpenBao Transit sign op or write the gateway's JWKS mount can forge tokens for ANY org/project with UNBOUNDED lifetime (the 90s TTL only constrains honest mints); the ESO-mounted AES sealing key for backend credentials is a second concentration.**
  - _Mitigation:_ Gateway-side hard lifetime ceiling: reject any EIA where exp-iat > 15m regardless of claims (S4), making forged long-lived tokens impossible and kid-removal-from-JWKS a real revocation channel; rapid key-rotation support + alerting on anomalous mint volume/unknown kid; STS→Transit sign access locked down and audited; JWKS ConfigMap/mount write-RBAC-restricted, integrity-protected, change-audited; gateway trust root scoped to the beads signing kid(s) only; AES sealing-key blast radius documented with a rotation runbook.
- **[HIGH] Removed or role-downgraded org members retain mint capability for up to 8h: human STS sessions do not re-check membership at token exchange (ceiling frozen at login), so offboarding without org-entitlement revocation leaves a departed member able to mint and read/write every reachable project for the session remainder.**
  - _Mitigation:_ GA BLOCKER (S3), not a runbook step: org-membership-change and role-change events wire to automatic session bulk-revoke (sec_offboard/org_bulk_revoke), AND the /sts/v0/token exchange re-checks membership for human sessions (as service sessions already re-check SP liveness) — removal severs mint within ONE exchange. Residual: a shorter human session TTL for the beads audience remains an open tuning question.
- **[HIGH] Org-wide token blast radius: until pins ship, any entitled org member's EIA reaches EVERY project in the org (pins enforceable at the gateway but unmintable at the STS).**
  - _Mitigation:_ GA GATE (S3): pin-aware STS intersection (grant beads:read#prj_<id>/beads:write#prj_<id> iff the bare base is in the session ceiling); getToken --project flag; recipes mint pinned scopes by default; gateway enforcement already exists (hasCapability). Pre-S3 operation is restricted to named design partners; GA does not ship without pins.
- **[HIGH] Cross-tenant reach on SHARED substrates (selfhosted dolt, sharedpg): many projects share one server; bd's own SET search_path rides the opaque splice invisible to the gateway, so the per-project SQL user's GRANTS — not search_path pinning — are the real boundary; one misprovisioned user with broad grants yields cross-tenant read/write.**
  - _Mitigation:_ Stated honestly in the contract (grants are the boundary; the gateway's startup-param pinning is a routing control, not the isolation mechanism). Automated grant-scope conformance check — the project's SQL user can touch only its own schema/DB, no cross-schema/global grants — run at project enrollment on shared substrates and as a recurring audit, with an escape test (a shared-substrate connection cannot read another project's schema) in CI; tenancy join (org→account, project∈account, workspace gate, RTBF) enforced before any backend dial; dedicated substrates (the live k8sdolt path, dedicatedpg-*) preferred for higher-isolation tenants — cross-tenant reach there is structurally impossible.
- **[HIGH] Compromise of the gateway itself (it sees every credential and every ledger byte in plaintext at the splice point).**
  - _Mitigation:_ Custody split preserved as contract: gateway holds NO backend root (Admin=nil — structurally cannot CREATE USER/GRANT); per-project backend creds AES-256-GCM-sealed with AAD=project_id|role, key ESO-mounted with a rotation runbook (S4); backend legs SPIFFE-mTLS with pinned CA, never skip-verify; inbound EIAs 90s TTL + 15m server ceiling so captured tokens age out fast; synchronous fail-closed mint audit at the STS gives a complete forensic ledger; per-session byte/duration metering keyed by account+project.
- **[HIGH] Weaponized workspace config beyond the host vector: a cloned repo's metadata.json could name an attacker-controlled credential helper or smuggle shell into helper args.**
  - _Mitigation:_ Helper persistence is allowlist-only (built-in binary names, initially ["gasworks"]; no shell; exact argv; strict param charset for org/project), gated by trust-on-first-use recorded OUTSIDE the repo keyed by workspace path + hash(helper, args, gateway-ref, database) — any change re-prompts via the attach ceremony; freeform credential commands remain env-only; the ':@/' token charset rejection and KindIdentity requirement stay as defense in depth; combined with the destination allowlist, a hostile repo can neither choose the binary, the argv, nor the destination.
- **[MEDIUM] PROXY-protocol source-IP spoofing: if the gateway accepts PROXY v2 headers from arbitrary peers on a public L4 listener, an attacker spoofs any source IP — defeating per-IP caps (128/IP) and poisoning audit/abuse attribution.**
  - _Mitigation:_ PROXY v2 accepted ONLY from an allowlist of trusted upstreams (the haproxy loopback NodePort source); headers from any other peer rejected/ignored with fallback to the real TCP source; no listener ever both accepts untrusted direct connections and honors PROXY headers; CI test that a direct client cannot spoof its attributed IP (S7 merge gate).
- **[MEDIUM] Public internet exposure of pre-auth MySQL/pg handshake parsers (DoS and parser-bug surface).**
  - _Mitigation:_ Retained pre-auth hardening: 8KiB handshake clamp (fits multi-KB JWTs), 10s handshake deadline, per-pod 4096 conns / 128-per-IP / 256 burst / 512 accepts-per-sec, per-connection panic containment, scrubbed uniform 'access denied'; PROXY v2 (above) makes per-IP caps meaningful; haproxy per-source stick-table retuned for public populations; load/abuse test of the pre-auth surface gates the public bind (S7).
- **[MEDIUM] Stolen-token replay window and the temptation to widen it: the S8 revalidation loop bounds PRINCIPAL revocation, NOT token theft — a stolen valid EIA maps to a live principal and passes revalidation, so TTL is the ONLY bound on replay.**
  - _Mitigation:_ The draft's 90s→15m TTL raise is DROPPED from the design: the wire token stays 90s at GA, mint load is engineered with caches/jitter/per-dial reuse instead. Any future raise is a separately-gated security decision requiring the destination allowlist (S2) and at-rest custody (S4) to be live, explicit security sign-off, and remains bounded by the gateway's 15m server-side ceiling; the spec states explicitly that revalidation bounds principal revocation (~5m) and TTL bounds token theft.
- **[MEDIUM] Revoked principal retains an already-open spliced session.**
  - _Mitigation:_ Pre-S8: the splice dies at token expiry (≤90s). Post-S8 (splice lifetime decoupled): periodic authz revalidation every 5m per live splice (session/SP liveness, account status, project membership, RTBF) severs live sessions in ≤5m; SP disable already propagates to NEW mints within one TTL via per-exchange liveness recheck; membership removal severs mint within one exchange (S3).
- **[MEDIUM] ACME/WebPKI dependency and cert-migration breakage: a lapsed gateway cert fails closed for every external client simultaneously; and swapping the live leaf can break EXISTING production clients that dial internal names the public cert cannot cover.**
  - _Mitigation:_ Client-inventory gate BEFORE any cert change (S1): enumerate live gateway clients and how each satisfies TLS; dual-cert SNI via GetCertificate keeps internal names on an internal cert until clients migrate; single secret owner (cert-manager-managed Secret; ESO leaf for gateway-tls deleted); GetCertificate doubles as renewal hot-reload; expiry alerting (30/14/7-day) + renewal runbook. Fail-closed on lapse is accepted as correct.
- **[MEDIUM] Durable bts_ static tokens remain a parallel long-lived credential path until sunset: static, revocation = row removal, no automatic expiry — any historic leak grants durable project access, undercutting the short-lived story.**
  - _Mitigation:_ Bounded NOW (S4), sunset later (S9): the gateway rejects non-expiring bts_ tokens; per-project bts_ acceptance flag defaulted OFF for newly-provisioned hosted projects (new projects cannot mint them); pre-GA inventory of existing bts_ per project with last-used telemetry; never emitted in any recipe, panel, or doc; full telemetry-gated sunset in S9.
- **[LOW] JWT-as-username is nonstandard wire usage: a future issuer could mint tokens that break the DSN (charset) or exceed handshake limits.**
  - _Mitigation:_ Frozen as contract: token must be base64url+'.' (no ':','@','/' — enforced client-side at gateway_credential.go:53), maximum length bounded well under the 8KiB pre-auth clamp; an STS contract test asserts minted beads EIAs satisfy both (S9).

## Performance

- Steady state target: mint rarely, connect once, splice bytes, translate nothing. Connect-time budget after S7: offline JWT verify + 2 indexed SELECTs + AES-GCM open + one backend dial — the AuthorizeEIA ListProjects + O(projects-per-account) linear scan is replaced by an indexed GetProjectInAccount point lookup BEFORE public exposure multiplies connect rate (bd CLI = new process + new connect per command).
- Mint-load is engineered, not TTL'd away (design change from the draft): the EIA TTL stays 90s; amortization comes from bd's per-process credential cache (to expiry−10s), the gasworks per-(org|product|scope) cache with 3-tier refresh skews, JITTERED early refresh spreading re-mints across the TTL window (never synchronized on the boundary), exponential backoff on STS 5xx/429, and serve-last-good-until-true-expiry through brief STS outages. The per-dial BeforeConnect connector reuses the cached token across pool dials and only re-execs the helper on expiry or 1045-invalidate.
- STS is on the data-plane critical path and is treated as such: an explicit STS availability + p50/p99 latency SLO with alerting is a GA gate (S4); the client-side jitter/backoff/serve-last-good behaviors above are REQUIRED bd/gasworks behaviors (S2), not suggestions, so an STS blip degrades to stale-token reuse within TTL rather than a fleet-wide thundering herd.
- Splice lifetime decoupled from token expiry in S8: replace SetDeadline(tc.ExpiresAt) — which today turns any operation >90s into a reconnect storm and kills long imports mid-flight — with idle timeout 30m + absolute cap 8h + 5m periodic authz revalidation (cheap indexed SELECTs per live splice). bd's existing pool (ConnMaxLifetime 1h) then actually works; until S8, short-lived CLI is the declared supported shape and the docs say so.
- Wire-path efficiency preserved and made tunable: InterpolateParams=true (no per-statement prepare round-trip), 5s connect timeout, 10s pooled read/write timeouts, per host:port:db circuit breaker — connect/read/write timeouts exposed as BEADS_DOLT_SERVER_*_TIMEOUT for public-internet latencies without changing tailnet defaults.
- Concurrency envelope: 1 inbound conn = 1 backend conn (no gateway pooling in v1); per-pod caps 4096 conns / 128 per IP / 512 accepts-per-sec / 256 burst; 2 replicas. PROXY protocol v2 from allowlisted upstreams (S7) is what makes the per-IP figures meaningful publicly; the haproxy per-source conn-cur 64 stick-table (the cause of the observed rejects) is retuned/retired at the same time.
- Known future component, explicitly deferred: a pooling tier (pgbouncer-style) if public connect concurrency approaches the per-pod cap; dedicated substrates isolate blast radius meanwhile.
- Latency SLOs for GA: p50 connect overhead with a cached token < 150ms over the client's raw TCP+TLS baseline to the gateway; p50 full mint path (getToken exchange) < 500ms; bd attach end-to-end < 5s on a warm login; STS SLO numbers themselves are an open sign-off item.

## Usability


### Golden path

1. Org owner (once per org): ensure the org has the beads entitlement and members; optionally create a workspace-bound service principal key for CI (beads_live_...).
2. User, first time on a machine: install bd + gasworks CLI via the panel's one-line installer (`curl -fsSL https://get.gascity.com/bd | sh`), which serves the tagged bd release containing the client changes and the pre-trusted default gateway entry.
3. `gasworks login` — opens the browser (device-code fallback via --device-auth for SSH boxes), explicit org choice, 8h DPoP-bound session (key in the OS keyring after S4). Panel caption: 'one-time per machine, sessions last 8 hours'.
4. Open the project in the beads console → Connection panel → click the ONE copy button: `bd attach "beads+dolt://gw.beads.gascity.com:3306/bd_prj_8baf2d1e07a932f1?org=acme&project=prj_8baf2d1e07a932f1"` (pg projects emit beads+pg://...:5432/...?schema=... once S6 ships; before that the panel shows the fail-closed 'endpoint rolling out' state).
5. Run the attach command in the working directory: it shows host + org + project (gw.beads.gascity.com is pre-trusted, so the golden path has no prompt; an unknown host requires explicit confirmation), preflights helper + login, makes a verification connect through the full credential ladder with Gateway semantics, ADOPTS the server's issue_prefix + project-identity UUID, writes .beads/metadata.json (helper/org/project — never the host) + key file, records trust-on-first-use, and prints 'Attached to acme/my-project (read-write). Try: bd list'.
6. `bd list`, `bd create`, everything — each new physical dial transparently presents a fresh short-lived EIA minted only for the trusted gateway; new terminals and future sessions need zero exports.
7. Teammate flow (metadata.json is designed to be committed): clone → first bd command detects well-formed-but-untrusted hosted config and prints exactly `gasworks login && bd attach --accept` → done.
8. CI: export GASWORKS_API_KEY=beads_live_... and BD_TRUST_WORKSPACE=1, run the identical attach command — getToken exchanges via /sts/v0/machine, trust is auto-accepted only because the API key is present; zero recipe divergence.
9. When anything fails: `bd doctor` re-runs the pipeline stepwise (workspace → helper → session → gateway-trust → DNS/TCP → TLS → auth → database) with per-layer PASS/FAIL, one remedial command per failure, and an out-of-band org↔project cross-check via the connection API.

### Error messages (named layer + one remedy)

- **bd invoked with no discoverable workspace (current 'no beads database found' exit, cmd/bd/main.go:1046-1052)** → no beads workspace here — for a hosted project, copy the `bd attach ...` command from your project's Connection panel and run it in this directory (or set BEADS_DIR).
- **Workspace references a gateway host not in the user-level trusted-gateway allowlist (ambient command, not attach)** → this workspace points at gateway 'evil.example:3306', which is not trusted on this machine — bd will not mint or send credentials to it. If this is intentional, review and trust it explicitly: bd attach <recipe URI>.
- **gasworks CLI missing, or no live session, when the credential helper runs (attach preflight or per-dial mint failure with expired-session stderr)** → cannot mint a credential: gasworks session missing or expired — run `gasworks login` (sessions last 8 hours), then retry. CI: set GASWORKS_API_KEY.
- **Helper exits non-zero with an STS 403 (entitlement or membership denial — distinct from expired session; helper stderr surfaced verbatim)** → credential mint refused: org 'acme' lacks the beads entitlement or your membership was removed — ask an org admin. (This is not a login problem; re-running gasworks login will not fix it.) [helper stderr follows verbatim]
- **gasworks getToken invoked with a --gateway host outside its allowlist (destination-aware mint refusal)** → refusing to mint a beads credential for unknown gateway 'evil.example' — trusted gateways: gw.beads.gascity.com. Add a gateway explicitly with `gasworks trust-gateway <host>` only if you operate it.
- **DNS resolution or TCP connect failure to the gateway host** → cannot reach gw.beads.gascity.com:3306 — check network/VPN/firewall. `bd doctor` will show which layer failed.
- **TLS verification failure (wrong/expired cert, corporate TLS interception middlebox)** → server certificate for gw.beads.gascity.com is not trusted by this system — if your network intercepts TLS, hosted beads cannot be used through it; bd never skips certificate verification.
- **Gateway pre-auth 1045 'TLS required' (client dialed without TLS because BEADS_DOLT_SERVER_TLS was unset/false)** → the beads gateway requires TLS and this connection attempted plaintext — set BEADS_DOLT_SERVER_TLS=true, or re-run `bd attach` which configures TLS correctly.
- **MySQL/pg auth error 1045 'access denied' after one silent cache-invalidate + re-mint retry** → access denied by the beads gateway — your session may have expired (`gasworks login`), your org 'acme' may lack the beads entitlement or your membership, your token may be pinned to a different project, or this project may belong to a different org — re-copy the Connection command. `bd doctor` cross-checks the org/project pairing.
- **Engine mismatch (gateway ErrEngineMismatch / pg-leg redirect message spliced back)** → this project's backend uses the other wire protocol — re-copy the `bd attach` command from the Connection panel; the project's backend changed since this workspace was attached.
- **pg-engine project whose gateway leg is not yet deployed (panel/API state, pre-S6)** → this project's connection endpoint is rolling out — local access for Postgres-backed projects is not yet available; no workaround recipe exists by design. Watch the panel for activation.
- **bd attach against a hosted database that has no provisioned project identity (provisioning-contract violation)** → this hosted project is missing its provisioned identity (_project_id) — this is a service-side provisioning error, not something to fix locally; contact support. bd will not invent an identity for a hosted database.
- **Workspace project-identity UUID does not match the server database (verifyProjectIdentity / pg equivalent)** → this workspace belongs to a different project than bd_prj_X on the server — refusing to write. Re-run `bd attach` in a fresh directory, or with --force-reattach after verifying the project id.
- **Persisted credential helper configuration changed since trust was recorded (TOFU hash mismatch)** → this workspace's credential configuration changed (helper/org/project/database) — review and re-trust: bd shows the exact command it wants to run and asks for confirmation once. CI: BD_TRUST_WORKSPACE=1 with GASWORKS_API_KEY set.
- **Fresh clone with committed well-formed hosted config but no local trust/key file (second-user flow)** → this repo is attached to hosted project acme/my-project but this machine hasn't trusted it yet — run: gasworks login && bd attach --accept
- **Read-only token used for a write (gateway served the read-only backend credential), or org billing hard-blocked** → write rejected: this connection is read-only — your token was minted --read-only, your role grants beads:read only, or the org is billing-blocked. Check `gasworks getToken beads --org acme --project prj_X` output scopes.
