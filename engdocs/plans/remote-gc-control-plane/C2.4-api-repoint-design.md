## 1. The core tension & recommendation (THE key decision)

### The tension

`Provision` writes `city.toml` inline at step 15 — a fresh add goes through the
surgical `config.AppendRigAndWriteSiteBindingsForEdit` and a re-add through
`config.WriteCityAndRigSiteBindingsForEdit` (`internal/rig/provision.go:384,386`).
The API side, however, has a contract every other `StateMutator` mutation honors
through `mutateAndPoke` (`cmd/gc/api_state.go:1753-1781`):

1. **disk snapshot** of `city.toml` + `.gc/site.toml` + agent scaffolds before the
   mutation (`captureConfigMutationSnapshot`, `api_state.go:1642-1727`);
2. the mutation writes disk;
3. **`refreshConfigSnapshot`** re-loads the composed config from disk and swaps it
   into controller memory — including a full `buildStores` rebuild
   (`api_state.go:1783-1801` → `update` `:617-676` → `buildStores` `:253-292`);
4. on refresh failure, **restore the disk snapshot** so disk never diverges from the
   memory the reconciler acts on (`:1766-1773`);
5. `markConfigMutationPending` + `configDirty` + `Poke` (`:1775-1779`).

G17 (`DESIGN-BRIEF.md:65`, `PHASE-2-HANDOFF.md:58`) requires the rig-create SUCCESS
signal only after `s.state.Config()` shows the rig **and** `state.BeadStore(rigName)
!= nil`. Both are products of step 3.

### The two options

**Option A — Provision runs AS `mutateAndPoke`'s mutate closure.** `mutateAndPoke`
never cared *how* its mutate closure writes disk: every existing mutation's closure
writes disk itself (e.g. `cs.editor.CreateAgent` inside the closure at
`api_state.go:1384-1386`). Provision's inline write is just a bigger closure. The
key realization that dissolves the "bypass" worry: **`mutateAndPoke`'s
snapshot/restore is not a write path — it is a disk-restore contingency for the
in-memory refresh failing after a successful write.** That is orthogonal to
Provision's own mid-write topology rollback (`internal/rig/provision.go:350-408`),
and the two compose cleanly:

- **Mid-provision failure** → Provision's `rollbackError`/panic-guard restores the
  topology snapshot itself; `mutateAndPoke` sees `mutate()` return an error and
  returns it **without** touching its own snapshot (`api_state.go:1762-1764` — no
  restore on mutate error). One restorer, no double-restore.
- **Provision succeeds, refresh fails** → `mutateAndPoke` restores the
  *pre-provision* `city.toml`/`site.toml` (its snapshot was captured before
  `mutate()` ran), no poke, no dirty flag — exactly what
  `TestControllerStateMutationRollsBackWhenRefreshFails`
  (`cmd/gc/api_state_test.go:1001-1049`) pins today.

**Option B — inject a `Deps.WriteConfig` seam** so the API's config write itself is
a `mutateAndPoke`-wrapped writer. Rejected on four grounds:

1. **It fires refresh + `Poke` mid-provision**, before the deferred `packs.lock`
   commit and the routes write (`provision.go:398-409`). The reconciler could tick
   against a config whose `routes.jsonl` does not exist yet — inverting the
   load-bearing write order (config → packs.lock → routes, plan invariant 1,
   `C2-EXTRACTION-PLAN.md:43-49`) as observed by concurrent readers.
2. **The seam signature is ugly and CLI-risky.** Step 15 *chooses* the writer from
   `reAdd` state computed inside Provision (`provision.go:371-391`), so
   `WriteConfig` would need `(fs, tomlPath, nextCfg, newRig config.Rig, reAdd bool)
   error` and the CLI closure would have to reproduce the surgical-append-vs-rewrite
   branch (Decision A1) byte-identically — a Deps + Provision + CLI change to a path
   whose contract is "byte-identical under 62 `TestDoRigAdd*` funcs"
   (`C2.2-provision-design.md:8-14`), for zero functional gain.
3. **Rollback entanglement.** A refresh failure inside step 15 would trigger
   `mutateAndPoke`'s restore *and* Provision's `rollbackError` restore over
   overlapping file sets, and Provision would then continue or abort with disk in a
   state neither snapshot layer fully owns.
4. **It re-creates the red-team's high finding.** Putting the must-succeed refresh
   behind an injected seam (or `PostProvision`) is where its error can get lost.
   Option A keeps refresh failure on `CreateRig`'s direct return path — surfaced to
   the HTTP client via `mutationError` (`internal/api/huma_types.go:227-241`),
   structurally impossible to swallow.

### Recommendation: **Option A**, with two refinements

```go
// CreateRig provisions a rig through internal/rig.Provision (Decision 7) and
// commits it to controller state through the standard mutateAndPoke handshake.
func (cs *controllerState) CreateRig(r config.Rig) error {
    // pre-steps: §2 (validate/resolve path, dup-name 409 guard)
    return cs.SerializeConfigWrite(func() error {          // refinement 1
        return cs.mutateAndPoke(func() error {
            // load for-edit cfg, register dolt config, build Deps — §2
            _, res, err := rig.Provision(deps, req)
            if err != nil {
                return err
            }
            return res.PostProvisionErr                    // refinement 2
        })
    })
}
```

- **Refinement 1 — serialize under the config-write lock.** Today's editor-based
  mutations serialize because every write happens inside `configedit.Editor`'s lock;
  Provision's `Append…ForEdit` write does not take that lock. Wrapping the whole
  `mutateAndPoke` call in `SerializeConfigWrite` (`api_state.go:1321-1323`, the same
  `editor.Do` lock the pack-import handlers use) prevents a concurrent
  `CreateAgent`/`UpdateRig` from interleaving with Provision's
  read-modify-append, and as a bonus closes the documented snapshot/restore race
  class (`api_state.go:1405-1409`) for this mutation. No deadlock risk: nothing
  Provision or its injected deps call takes the editor lock.
- **Refinement 2 — surface `PostProvisionErr`.** The landed core captures a non-nil
  `Deps.PostProvision` error in `ProvisionResult.PostProvisionErr` instead of
  discarding it (`internal/rig/deps.go:104-108`, `provision.go:431-438`) — the fix
  for the red-team's high finding. In C2.4 the API `PostProvision` closure always
  returns nil (§2), so this is defensive; in C4's async path the goroutine maps it
  to `events.RequestFailed`.

**G17 verification under Option A.** When `mutateAndPoke` returns nil:
`refreshConfigSnapshot` has already loaded the post-write config and run
`cs.update(nextCfg, sp)` (`api_state.go:1799`), which synchronously swaps `cs.cfg`
and rebuilds `cs.beadStores` including the new rig (`update` `:651-659`,
`buildStores` `:267-291` — the rig has a non-empty path, so it is not skipped by the
unbound-rig guard at `:274`). Therefore, at the moment `CreateRig` returns —
which is when `humaHandleRigCreate` emits its 201 and when C5's future SUCCESS event
will be emitted — both G17 conditions hold: `Config()` shows the rig and
`BeadStore(rigName)` is registered. Store *liveness* (a deferred managed-Dolt init
finishing) is deliberately NOT part of this barrier; that is C4's async-status
concern (see §7 R6). Note the ordering guarantee the old code expressed as
"initialize store before publishing" (`initializeRigStoreForCreate` running before
`mutateAndPoke`) is preserved: `Deps.InitStore` runs inside `mutate()`, strictly
before the refresh that publishes the rig — pinned by
`TestControllerStateCreateRigInitializesStoreBeforePublishing`
(`api_state_test.go:960-999`).

## 2. `controllerState.CreateRig` rewrite

`api_state.go` is `package main` (`cmd/gc/api_state.go:1`), so every `cmd/gc`
function the CLI wrapper injects is directly reachable — the Deps table is nearly
the CLI's (`cmd/gc/cmd_rig.go:242-307`), diverging only where the caller context
differs.

### Pre-steps (before `SerializeConfigWrite`)

1. **Path validation + resolution.** `rigPath := strings.TrimSpace(r.Path)`; if
   empty, return `fmt.Errorf("%w: rig path is required", configedit.ErrValidation)`
   (→ HTTP 400 via `mutationError`; `RigCreateInput.Path` has `minLength:"1"` at
   `internal/api/huma_types_rigs.go:28`, so only whitespace-only input reaches
   this — today that silently skipped store init at `api_state.go:1523-1525`, a
   hole, not a contract). Then `rigPath = resolveStoreScopeRoot(cs.cityPath,
   rigPath)` (`cmd/gc/main.go:1339-1349`) — **cityPath-relative, never CWD-relative**,
   exactly what `detectRigDefaultBranch` does today (`api_state.go:1512`) and what
   `TestControllerStateCreateRigDetectsDefaultBranchForRelativePath`
   (`api_state_test.go:911-951`) pins (a same-named rig in the daemon CWD must NOT
   win). This satisfies `validateRequest`'s absolute-path requirement
   (`internal/rig/deps.go:142-147`). The CLI's `resolveRigAddPath` dot-relative
   branch stays CLI-only, per the brief (`DESIGN-BRIEF.md:60`).
2. **Duplicate-name 409 guard** — port verbatim from `initializeRigStoreForCreate`
   (`api_state.go:1527-1536`): under `cs.mu.RLock`, scan `cs.cfg.Rigs` for
   `r.Name` and return `fmt.Errorf("%w: rig %q", configedit.ErrAlreadyExists,
   r.Name)`. This preserves the API's 409-on-existing-name contract. Without it,
   Provision's re-add semantics would make same-name+same-path an idempotent
   success and same-name+different-path a plain error (`provision.go:110-114`) →
   HTTP 500. Config-level re-add idempotency for the API is explicitly deferred to
   the C4/G13 `request_id` machine (`G13-request-id-state-machine.md:199-204`),
   which owns retry semantics.

### Inside `mutateAndPoke`'s closure

3. **Load the for-edit config** — `loadCityConfigForEditFS(fsys.OSFS{},
   filepath.Join(cs.cityPath, "city.toml"))` (`cmd/gc/cmd_agent.go:167-176`),
   mirroring the CLI wrapper (`cmd_rig.go:224`). **NOT `cs.cfg`**: the controller
   snapshot is the composed/expanded runtime config (builtin packs, patches,
   site bindings applied); writing `city.toml` from it would bake expansions into
   the file. The retired editor path made the same choice internally
   (`configedit.Editor.Edit` loads raw). Note `config.Load` skips include/pack
   expansion by design (`internal/config/config.go:5127-5130`) — this is what keeps
   the refresh-rollback test's failure point intact (§4).
4. **Dolt-config registration pairing** — same gate and same lexical scope as the
   CLI wrapper (`cmd_rig.go:233-236`): `if cityUsesBdStoreContract(cs.cityPath) &&
   cityDoltConfigHasLifecycleFields(cfg.Dolt) { registerCityDoltConfig(...); defer
   clearCityDoltConfig(...) }` wrapping the Provision call (plan invariant 3,
   `C2-EXTRACTION-PLAN.md:51-54`).

**The API `PostProvision` closure** mirrors the CLI tail (`cmd_rig.go:259-299`)
minus the two pieces the controller context replaces or forbids:

- KEEP (rig-local infrastructure, functionally load-bearing):
  `ensureGitignoreEntries`, `hooks.InstallWithResolver` (gated on
  `pc.Cfg.Workspace.InstallAgentHooks`), the `config.LoadWithIncludes` reload +
  `ResolveFormulas`, and `writeBeadsEnvGTRoot` (agents' `bd` invocations read
  `GT_ROOT` from `.beads/.env` — an API-created rig without it is subtly broken).
  Warnings go to `log.Printf("api: rig create: ...")` instead of the CLI's
  stderr prints (consistent with `api_state.go`'s existing best-effort logging,
  e.g. `:157`, `:686`).
- DROP `installBeadHooks` (adopt-only; `Adopt` is always false).
- DROP `rigReloadControllerConfig` — **G17 explicitly forbids it**: a controller
  dialing its own control socket mid-request self-deadlocks (`DESIGN-BRIEF.md:65`).
  Its job (making the controller see the rig) is done by `mutateAndPoke`'s refresh,
  which runs after Provision returns.
- DROP `rigWaitForStoreAccessible` (`cmd_rig.go:293-297`) — a synchronous HTTP
  handler must not block up to `rigDeferredStoreInitWait` (30s, `cmd_rig.go:24`).
  G17's barrier is store *registration* (satisfied by refresh → `buildStores`),
  not liveness; the deferred-init liveness wait becomes C4's async-status concern.
- The closure returns nil always (all its failures are warn-and-continue), matching
  the CLI contract (`deps.go:106-108`).

**The ProvisionRequest** built from the handler's `config.Rig`
(`internal/api/huma_handlers_rigs.go:73-78` shows the only fields the wire can set):

```go
rig.ProvisionRequest{
    Name:          r.Name,          // required by the wire (minLength:1)
    Path:          rigPath,         // resolved absolute (pre-step 1)
    Prefix:        r.Prefix,        // Provision lowercases/derives (provision.go:117-126)
    DefaultBranch: r.DefaultBranch, // Provision trims + probes when empty
    Includes:      nil,             // no wire field yet; C4 may extend RigCreateInput
    StartSuspended: false,          // no wire field
    Adopt:          false,          // no wire field
}
```

The returned `config.Rig` from Provision is discarded (the handler builds its 201
body from the input name only, `huma_handlers_rigs.go:83-86`); C4's async result
event will carry it.

## 5. The byte-identical local-vs-API artifact test

**File:** `cmd/gc/rig_provision_parity_test.go` (must be `package main`: it needs
both `doRigAddWithResult` and `controllerState`, and G12 names exactly this pairing,
`DESIGN-BRIEF.md:60`).

**Test:** `TestRigAddLocalAndAPIProduceIdenticalArtifacts`.

```
Env:   t.Setenv("GC_BEADS", "file"); t.Setenv("GC_DOLT", "skip");
       t.Setenv("GC_BEADS_SCOPE_ROOT", "")        // mirrors api_state_test.go:853-854,885-886
Setup: two temp cities A and B with byte-identical city.toml
       ([workspace] name = "parity-city"), and a pre-created empty rig dir
       <city>/repo in each (inside the city → no site-binding path divergence;
       no .git → the probe is skipped identically on both paths).
CLI:   doRigAddWithResult(fsys.OSFS{}, cityA, filepath.Join(cityA,"repo"),
           nil, "", "", "", false, false, io.Discard, io.Discard)   // cmd_rig.go:215
API:   cs := newControllerState(ctx, cfgB, runtime.NewFake(), events.NewFake(),
           "parity-city", cityB)                                    // api_state.go:119
       cs.CreateRig(config.Rig{Name: "repo", Path: filepath.Join(cityB, "repo")})
```

**Comparison set** (read each from both cities; normalize by replacing the absolute
`cityA`/`cityB` strings with a `<CITY>` placeholder before comparing, since the

## 7. RISKS

- **R1 — mutateAndPoke integration is the load-bearing novelty.** The composition
  argument in §1 (Provision's rollback owns mutate-time failures; mutateAndPoke's
  snapshot owns refresh-time failures; `api_state.go:1762-1764` proves no
  double-restore) must hold in code review. A subtle regression here (e.g. calling
  `Provision` outside the closure and `mutateAndPoke(func() error { return nil })`
  after) would silently break the rollback tests' contract. The §4 table's two
  rollback pins (`:1001`, `:3092`) are the tripwires — do not restructure them.
- **R2 — refresh-failure orphans (new residual, accepted).** If Provision fully
  succeeds and `refreshConfigSnapshot` then fails, `mutateAndPoke` restores
  `city.toml`/`site.toml` but NOT `routes.jsonl`, `packs.lock`-committed pack
  materializations, or the initialized `.beads` store — stale routes and a
  retry-poisoning store remain (the red-team's routes-not-in-snapshot and
  retry-poison findings, medium). Same class as the pre-existing CLI gap;
  disposition: accept in C2.4, route cleanup through C4's `request_id` state
  machine (`G13-request-id-state-machine.md`), which must treat "failed after
  store init" as a distinct resumable state.
- **R3 — packs/routes-in-API behavior change.** A hosted controller's
  `POST /rigs` now composes packs (including root-pack *default* rig imports the
  operator may not expect), writes routes for every rig, and normalizes scope
  files. This is Decision 7's intent, but it widens the API write surface: new
  failure modes, longer handler latency (beads init + pack resolution under the
  §1 config-write lock), and new on-disk artifacts on the server. Mitigations:
  the §5 parity test makes the shape explicit; latency is bounded in C2.4 by
  dropping the 30s store wait (§2); C4 moves the whole thing async behind 202.
- **R4 — coarse error mapping (follow-up).** Provision's untyped errors map to
  HTTP 500 via `mutationError`'s default arm (`huma_types.go:238-239`); only the
  pre-step dup-name guard yields 409 and the empty-path guard 400. Prefix
  collisions and adopt-class violations *should* be 4xx. Fix belongs in C3/C4
  (typed error classes on `internal/rig` errors), not in string-matching here.
- **R5 — import cycle: none, confirmed.** `controllerState` lives in `package
  main` (`cmd/gc/api_state.go:1`), which already imports `internal/rig`
  (`cmd/gc/cmd_rig.go:19`). `internal/api` never imports `cmd/gc`; it reaches
  provisioning only through the `StateMutator` interface value it is handed
  (`internal/api/state.go:277,309`). The `internal/rig` package imports neither
  (`internal/rig/deps.go:3-10`).
- **R6 — PostProvision-error surfacing for the API 202 path (red-team, forward).**
  C2.4's sync path returns `res.PostProvisionErr` from the closure (defensive;
  nil today). C4's async goroutine MUST map both a Provision error and a non-nil
  `PostProvisionErr` to `events.RequestFailed` with `request_id` (G20), and must
  keep the provisioning goroutine off the config-write lock for the clone stage —
  the §1 shape supports this by staging the clone *before* entering
  `SerializeConfigWrite`. Also inherited: G17's barrier here is store
  *registration*; C4's status machine owns liveness (`rigWaitForStoreAccessible`
  semantics) so a deferred managed-Dolt init is reported as `provisioning` rather
  than blocking the handler.
- **R7 — lock-hold duration.** Provision under `SerializeConfigWrite` serializes
  all config mutations behind beads init (managed-Dolt ready-wait,
  `beads_provider_lifecycle.go:309-318`). Single-operator control planes won't
  notice; C4's per-rig-name lock + async split (G15/G14) is the real fix. Do not
  "fix" it in C2.4 by dropping the lock — that reopens the read-modify-append race
  (§1 refinement 1).
