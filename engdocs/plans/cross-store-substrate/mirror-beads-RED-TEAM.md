# Mirror Beads — Red-Team Findings (37 survived of 44, 7 refuted)

Fable red-team workflow (50 agents: 6 attack dimensions x verify, refute-by-default). Ranked by adjusted severity. Verdicts checked vs design text + code.

## [1] HIGH · CONFIRMED · false-ready-classification
**A waits-for edge through a mirror is a no-op blocker on bd: is_blocked's waits-for branch gates on the TARGET's open children, and a mirror gate bead has none — false-ready despite an open mirror**

- **Scenario:** The guard covers all of beads.IsReadyBlockingDependencyType = {blocks, waits-for, conditional-blocks} (gascity internal/beads/beads.go:189-199). But bd's is_blocked mark template treats waits-for via waitsForGateBlockedSQL (beads issueops/blocked_state.go:22-57, used at :176-181): the dependent is blocked only while the TARGET has open parent-child children. A mirror is a childless gate row, so DepAdd(dependent, mirror, 'waits-for') classifies cleanly to depends_on_issue_id, yet the mark pass computes is_blocked=0 -> dependent appears in bd ready while the remote fanout spawner is open. markDirectBlockedSource also only handles blocks/conditional-blocks (domain/db/dependency.go:150-153). MemStore (memstore.go:303-312) and SQLiteStore (sqlite_store.go:922-928) block waits-for on the target's OWN status, so the mirror 'works' there — the design re-creates the exact MemStore/bd divergence it claims to end (SS6b). The escape hatch of minting the mirror edge as 'blocks' is also wrong: waits-for releases when the remote spawner's CHILDREN satisfy the gate (all-children/any-children), not when the spawner closes; the SS3 sync engine only mirrors the remote bead's status+close_reason, so a blocks-mirror stays open until spawner closure -> stuck-blocked. Either encoding violates SS1's core invariant ('to the local bd engine, a mirror is indistinguishable from any local blocker').
- **Design ref:** mirror-beads-DESIGN.md SS1 invariant, SS2 guard scope, SS3 sync; beads internal/storage/issueops/blocked_state.go:22-57,176-181
- **Fix:** Split ensureBlockingDependencyVia by dep type. Cross-store blocks/conditional-blocks: mirror + same-type local edge, as designed. Cross-store waits-for: hard error in v1/v2 ("cross-store waits-for unsupported: bd gates waits-for on the target's children, which a status-mirror cannot represent") — fail-loud, consistent with SS2's philosophy — and exclude waits-for from the SS6b auto-convert sweep (emit the warning event, do not convert). Document in SS2/SS3 that mirror sync (status+close_reason) is only a valid stand-in for target-own-status semantics. If cross-store waits-for is later required, it needs a distinct mirror kind whose sync predicate referral-evaluates the remote gate condition (open children per all-children/any-children metadata) and mints the local edge as "blocks" against that gate-mirror.

## [2] HIGH · CONFIRMED · false-ready-classification
**Mirror indirection breaks drain's member-to-item-root projection and its never-release guard: the item workflow ends up blocked on a source-member mirror that drains never release — permanent stuck-blocked in the design's own SS5 walkthrough topology**

- **Scenario:** drainProjectedBlockerIDs (gascity internal/dispatch/drain.go:590-625) reads the member's deps and maps dep.DependsOnID through rootByMember/manifestMembers, both keyed by MEMBER ids; anything unrecognized is embedded as a raw blocker on every item-workflow bead, and the code explicitly guards in-manifest members because 'drains do not close source members, so that edge would never release'. Once cross-store member edges are mirror-backed (SS4 'any cross-rig sling wiring', SS7 v2 all creation sites), the stored edge is r2-77 -> r2-90x (mirror of r1-42). r2-90x is neither a manifest member nor a member with a root, so the projection embeds the MIRROR as a raw blocker on r2-77's item workflow: the workflow now waits for r1-42 (a drain SOURCE member) to close — which the drain never does — instead of waiting on r1-42's item-root workflow. The never-release guard is bypassed because it matches member ids, not mirrors of member ids: permanent stuck-block, no error. The pre-mirror variant is as bad: a raw cross-store member edge lands external in the member's bd store and is invisible to DepList (finding 3), so drain never sees the edge at all -> item roots wired with NO dependency -> false-ready ordering between units. SS5 step 1 ('Drain projects the manifest edge') has no working data source in either world.
- **Design ref:** mirror-beads-DESIGN.md SS4 Creator, SS5 steps 1-2, SS7 v2; gascity internal/dispatch/drain.go:590-625 (drainProjectedBlockerIDs, in-manifest guard comment)
- **Fix:** Add a mirror-transparency requirement for member-edge readers to the design (new §4/§5 clause) and implement it at the three drain read sites: when dep.DependsOnID is not in rootByMember/manifestMembers, Get the bead from the member's store and, if Metadata["gc.mirror_of"] is set, substitute the remote id before the member-keyed checks in drainProjectedBlockerIDs, repairDrainWorkflowSourceMemberDeps, and orderDrainMembersByDependencies (drain.go:590/654/776). This restores the intended projection (mirror-of-in-manifest-member → blocker's item root when materialized, never-release guard otherwise; genuine out-of-manifest mirrors still embed raw and release via sync). Also rewrite SS5 steps 1-3 to state the drain-topology release signal is the blocker's item-root close, not member close.

## [3] HIGH · CONFIRMED · sync-loss-reopen
**Reopen fast path is structurally dead: the in-memory index holds only OPEN mirrors, but a reopen event by definition targets a remote whose mirror is CLOSED**

- **Scenario:** Design §3 says the index is 'hydrated at startup by ListByMetadata(...) per store for open mirrors, maintained on mirror create/close'. Ordering: (1) remote X closes, fast path closes mirror M and evicts it from the index (or controller restarts and hydration skips M because ListByMetadata defaults to non-closed beads — gascity internal/beads/memstore.go ListByMetadata doc: 'returns non-closed beads ... by default'; bdstore.go:2372 same). (2) Remote X reopens; the bead.updated event arrives; index lookup for X finds NOTHING; the event is silently dropped. Every reopen therefore misses the fast path 100% of the time — the only mirrors the fast path can 'reopen' are open ones (a no-op). Reopen handling degrades in ALL cases to the tick re-verify pass, which is itself filtered by 'still has open dependents' and 'age cutoff' (§8), leaving the permanent holes in the other findings. The §3 claim of event-latency reopen propagation is unimplementable as specified. Side gap: hydration needs a metadata PRESENCE query ('has key gc.mirror_of, any value'); MemStore/bd metadata filters are exact k=v matches, so hydration also needs a constant marker key.
- **Design ref:** mirror-beads-DESIGN.md §3 (event-driven fast path, index 'for open mirrors') vs §8 reopen staleness; gascity internal/beads/memstore.go:~370 (ListByMetadata non-closed default), internal/beads/bdstore.go:2372
- **Fix:** Amend §3: (1) the index must hold every mirror whose dependents are not all closed, regardless of mirror status — add on create, flip a status flag (do not evict) on close, evict only via orphan cleanup when all dependents are closed; (2) hydrate with a constant marker (e.g. metadata "gc.mirror"="true" registered in beadmeta, or a constant label via ListByLabel) passing beads.IncludeClosed, since metadata filters are exact k=v with no presence semantics; (3) reword §8 so the age cutoff on the re-verify pass applies only as a backstop for index-evicted (all-dependents-closed) mirrors, keeping indexed closed mirrors re-verifiable indefinitely.

## [4] HIGH · CONFIRMED · sync-loss-reopen
**Permanent closed-mirror divergence: remote reopens after all dependents closed; the 'closed mirrors with open dependents' re-verify filter never fires again; any later reuse of the mirror is instant false-ready**

- **Scenario:** Ordering: (1) mirror M in store B for remote X, dependent D. (2) X closes → M closes → D ready → dispatched → D closes. (3) X reopens (`bd reopen` — work found defective); the event is dropped (finding 1) or missed (finding 5). (4) Reconcile re-verifies only closed mirrors WITH open dependents — M has none — so M is never referral-read again. M stays closed while X is open: divergence is now permanent, silently. (5) Later a retry/new workflow wires a new dependent D2 on blocker X: ensureBlockingDependencyVia's dedup (§4) finds M via ListByMetadata and REUSES it — the creation-time referral read (§4 'Creation-time status') happens only on create, never on reuse. D2's edge binds to closed M → is_blocked recompute leaves D2 unblocked → D2 dispatches while its blocker is open. Amplifier grounded in code: drain re-wires blockers onto every workflow bead each pass (ensureDrainWorkflowBlocksOn loops listByWorkflowRoot and calls ensureBlockingDependency, internal/dispatch/drain.go:~630-650, control.go:284), so later-created retry/step beads keep acquiring edges to the stale closed mirror. This is exactly the false-ready class the design exists to kill, manufactured by its own dedup path.
- **Design ref:** mirror-beads-DESIGN.md §3 (catch-up reconcile filter), §4 (Dedup, Creation-time status); gascity internal/dispatch/drain.go:~630 ensureDrainWorkflowBlocksOn, internal/dispatch/control.go:284 ensureBlockingDependency
- **Fix:** In §4, replace "Creation-time status … once at creation" with wiring-time status: ensureBlockingDependencyVia must referral-read the remote and re-sync the mirror's status (reopen if remote open; close with the remote's reason if closed) on EVERY mirror acquisition — dedup reuse as well as create — before adding the new edge. (Creation and wiring are both write-path operations, so the foreign read is already sanctioned by the design's own words.) Corollary one-liner for the same section: the duplicate-collapse pass must re-sync the surviving mirror's status before repointing edges to it. This makes divergence on a dependent-less closed mirror inert (corrected at next reuse) and closes the instant false-ready, without widening the reconcile scan.

## [5] HIGH · CONFIRMED · dedup-collapse-crashsafety
**'Repoint' is two unfenced bd operations; remove-before-add ordering or a crash between them flips the dependent ready mid-collapse, and non-drain sites have no re-wiring pass to recover**

- **Scenario:** The design pins 'repoint before close' (§8) but never pins the ordering INSIDE repoint. bd recomputes is_blocked synchronously inside the DepRemove transaction (beads issueops/dependencies.go:745, domain/db/dependency.go:242). If collapse (or any implementer following the doc) removes D→M2 before ensuring D→M1, D has zero open blockers for the gap between the two commits → is_blocked=0 → the dispatcher can claim D in that gap with no crash at all. With a crash between the ops, the state is durable: D has NO mirror edge. Nothing recovers it outside drain: duplicate-collapse only repoints EXISTING edges; orphan cleanup then sees both mirrors with 'blocking edges gone' and closes them (§4), cementing the loss; drain re-projection would re-wire (drain.go:156 re-runs each tick) but only while the drain control bead is still open — v2 creation sites (sling wiring, API/CLI DepAdd, §7) are one-shot with no re-runner → permanent false-ready. Additional trap: bd dep Insert ERRORS when an edge to the same target exists with a different type ('already exists with type %q', dependency.go:106) — repointing a waits-for/conditional-blocks edge onto a winner that already carries a blocks edge from another site aborts the collapse mid-sequence, leaving mixed partial states every tick.
- **Design ref:** §8 'must be idempotent and crash-safe (repoint before close)' + §4 Dedup; grounded: beads internal/storage/issueops/dependencies.go:745, beads internal/storage/domain/db/dependency.go:96-108,242, gascity internal/dispatch/control.go:284, gascity internal/dispatch/drain.go:156
- **Fix:** Amend §4 (Dedup) and §8 to pin the repoint micro-ordering and type-conflict rule: for each edge (X → loser, type T), (1) FIRST ensure (X → winner) exists — if DepAdd fails with bd's "already exists with type" conflict, treat any existing ready-blocking-type edge on (X → winner) as satisfying the repoint (never remove-first to clear the conflict, despite bd's error-message advice); (2) only after the winner edge is confirmed present, remove (X → loser); (3) only close a loser mirror once it has zero blocking in-edges. This makes every intermediate state (including any crash point) fail-closed: the dependent always holds at least one open blocker edge, and an interrupted collapse re-converges on the next reconcile tick. Optionally add the §8 audit-test sweep assertion that no open remote blocker known to a mirror (gc.mirror_of) has a dependent with zero local blocking edges, as a backstop detector.

## [6] HIGH · CONFIRMED · dedup-collapse-crashsafety
**The dedup/reuse lookup is unspecified on closed mirrors, and both possible ListByMetadata behaviors are defective: open-only → unbounded mirror minting per drain tick; include-closed → reuse of a stale closed mirror after remote reopen without any referral-read**

- **Scenario:** Every gascity store's ListByMetadata excludes closed by default and returns newest-first (memstore.go, sqlite_store.go:1005). Branch (a), guard uses the default: remote r1-42 closes → sync closes mirror M → dependent becomes ready but the drain control bead is still open, so ensureDrainRowDependencyProjection re-runs every dispatcher pass (drain.go:156); the guard now finds NO open mirror for r1-42 → §4 creation path referral-reads the remote (closed) → creates ANOTHER already-closed gate bead and a new dep row (ensureBlockingDependency's existence check at control.go:284 is per-target-id, so each fresh mirror id passes) — one new bead + dep row per tick, real Dolt writes accumulating exactly the per-clone churn bd's blocked_state.go:134 discipline exists to avoid, until the drain finishes. Branch (b), guard passes IncludeClosed: remote closed → M closed → drain completes → weeks later the remote is REOPENED for rework and a new convoy wires D2 against r1-42; the guard finds M (closed), reuses it — §4 specifies the referral-read only on the CREATE branch, not the reuse branch — DepAdd(D2, M) binds D2 to a closed blocker → D2 is false-ready from birth while the remote is open. Rescue is the closed-mirror re-verify tick, but M's age is arbitrary → past the §8 age cutoff → permanent.
- **Design ref:** §4 'Dedup: key = (gc.mirror_of, local store). Guard with ListByMetadata before create' + §4 'Creation-time status'; grounded: gascity internal/beads/sqlite_store.go:1004-1013, gascity internal/beads/memstore.go ListByMetadata (SortCreatedDesc, IncludeClosed opt-in), gascity internal/dispatch/drain.go:156, gascity internal/dispatch/control.go:284
- **Fix:** Amend §4 of mirror-beads-DESIGN.md in three sentences: (1) the dedup lookup MUST pass IncludeClosed (and WithBothTiers) so closed mirrors satisfy the (gc.mirror_of, local store) key — note this deliberately diverges from the ensureDrainUnitConvoy pattern; (2) make ensureMirror an ensure on BOTH branches: on reuse, perform the same referral-read as the create branch and reconcile the mirror's status (reopen if the remote is open, align close reason if closed) before DepAdd; (3) key the §8 closed-mirror re-verify age cutoff to last-dependent-wired/last-sync-touch (or explicitly reset the age window whenever a blocking edge is wired to a closed mirror), so a rewired stale mirror is always eligible for re-verification.

## [7] HIGH · CONFIRMED · composition-with-drain
**Mirror double-wires with the ExternalDeps channel at item-root creation: raw foreign edge + mirror edge coexist, and the fail-closed raw edge stalls the drain forever even after the mirror releases**

- **Scenario:** Separate-context drain on a split city (graph=sqlite). Member mc-5 has an out-of-manifest blocking dep mc-99 (open, in the Dolt work store). expandDrain row wiring: (1) ensureDrainItemRoot passes blockerIDs=[mc-99] into molecule.Instantiate as ExternalDeps (drain.go:970-974); drainWorkflowExternalDeps fans it to EVERY step (drain.go:987-1016) and Instantiate embeds raw 'blocks -> mc-99' on each step bead via b.Needs / store.DepAdd (molecule.go:550-559, 713) — this channel is NOT in the design's §4 creator inventory and bypasses ensureBlockingDependencyVia entirely. (2) ensureDrainRowDependencyProjection then routes through Via, creates mirror gcg-M in the graph store, and adds 'blocks -> gcg-M' on the same workflow beads. Result: every step carries BOTH edges. SQLite Ready treats the absent mc-99 as not-closed (fail-closed LEFT JOIN, sqlite_store.go:923-929), so when a worker closes mc-99 and the controller closes gcg-M, the raw mc-99 edge still blocks every step. The item workflow never dispatches, completeDrain loops ErrControlPending forever (drain.go:353-354), the convoy never drains. The mirror added an edge but nothing removes the pre-existing raw one (repairDrainWorkflowSourceMemberDeps only strips manifest-member targets, drain.go:668-679; external-dep auto-conversion is explicitly deferred to v2, §7). The same permanent wedge hits every drain resumed from a manifest persisted by a pre-mirror build.
- **Design ref:** Design §4 'Creator: ... Today that is the drain projection (ensureDrainRowDependencyProjection, drain.go:543) and ensureBlockingDependency (control.go:284)' — misses drain.go:970-974 (molecule.Instantiate ExternalDeps) and molecule.go:550-559/713; §2 creation guard claims to 'wrap the blocking-edge writer' but there are two writers
- **Fix:** Route the Instantiate channel through the same mirror translation: in ensureDrainItemRoot (both call sites, drain.go:137/:499), before building ExternalDeps, map each blockerID whose prefix differs from the workflow store's prefix to its mirror id via the §2 ensureMirror helper (create-or-dedup mirror, substitute mirror id), so molecule.Instantiate only ever embeds store-local targets; amend §4's creator inventory to include the molecule.Instantiate ExternalDeps/b.Needs channel (drain.go:970-974, molecule.go:550-559). For drains resumed from pre-mirror manifests, extend repairDrainWorkflowSourceMemberDeps (or ensureDrainRowDependencyProjection) to DepRemove raw blocking edges on workflow beads whose target prefix is foreign to the store once the superseding mirror-backed edge exists.

## [8] HIGH · CONFIRMED · composition-with-drain
**§5 cross-rig walkthrough cannot execute through the actual drain, and a literal implementation mirrors the wrong object: source members that drains never close, producing permanently stuck-blocked members where today's flow is benign**

- **Scenario:** (a) Reachability: drain resolves members over exactly [graph primary, ONE scope-local work store] (drain.go:86,201; runtime.go:64-70; dispatcherControlStores cmd_convoy_dispatch.go:507-525). What closes is r1-42's item ROOT (gcg-), already wired root-to-root in ONE store by the existing projection. So after the drain succeeds, r2-90x stays open and r2-77 stays is_blocked=1 in rig-2's bd ready indefinitely — invisible to sling/gc hook --claim/any member-level dispatch — where today the depends_on_external row is a harmless no-op because ordering is enforced by the graph-store projection. - **Design ref:** Design §5 steps 1-3 (cross-rig convoy walkthrough) and §4 creator claim vs drain.go:590-626 (drainProjectedBlockerIDs root-projection), drain.go:266/292-299 (cross-store member rejection), runtime.go:64-70 (single WorkStore)
- **Fix:** Rewrite §4/§5 to match actual drain semantics and re-scope v1: (1) state that the drain's cross-store blocking edge has the graph-resident item-workflow bead as dependent and an out-of-manifest work-store bead as blocker, so ensureBlockingDependencyVia creates the mirror in the GRAPH store (per §1's dependent's-store rule) mirroring the foreign work bead — whose closure by its owning workflow/worker is a real release path; (2) state that in-manifest member-to-member edges get NO mirror: the existing item-root projection already enforces that ordering, and member-keyed blocking edges are the stall class repairDrainWorkflowSourceMemberDeps deletes; (3) replace the two-rig-member walkthrough with either an explicit v1 scope limit ('drain members must resolve in the scope's single work store; foreign-store blockers may only appear as out-of-manifest deps') or an explicit work item adding multi-store drain-member resolution (opts.WorkStore → member-store list plus reservation/dep-probe routing); (4) drop the reference to the nonexistent drainMemberOwningStore and name the real detector (blocker prefix ≠ dependent store's IDPrefix via storeref.PrefixOwner).

## [9] HIGH · PARTIAL · semantic-coverage-gaps
**gc.outcome — the field gascity's conditional/failure machinery actually consumes — is never synced to the mirror, so cross-store outcome decisions invert**

- **Scenario:** The design justifies reason-sync with 'conditional-blocks semantics key off the blocker's close reason' (§1). In this codebase they do not: bd's is_blocked treats conditional-blocks identically to blocks (blocked_state.go:152,197 — unblocks on ANY close), bd's IsFailureClose (beads internal/types/types.go:923-937) has ZERO production callers, and every real failure decision in gascity reads gc.outcome/gc.failure_class/gc.failure_reason METADATA (beadOutcomeFailed, gascity internal/dispatch/runtime.go:696-718; classifyRetryAttempt, dispatch/retry.go:248-280; tally voter outcomes, tally.go:75). The mirror syncs status+reason only and explicitly never syncs metadata (§1 'What it is NOT'). Ordering: member r2-77 (rig-2) has a cross-store conditional-blocks/blocks edge on r1-42 (rig-1); r1-42 closes with gc.outcome=fail. The mirror in rig-2 closes carrying at best a reason string. Any consumer that follows r2-77's dep to ask 'did my blocker fail?' Gets the MIRROR: beadOutcomeFailed(mirror) returns false (no gc.outcome, no gc.on_fail on the mirror), i.e. a FAILED remote blocker reads as not-failed — inverted decision — and 'B runs only if A FAILED' can neither be enforced nor even audited from the local store. Nothing on the ready path enforces the conditional either (bd unblocks conditional-blocks on any close), so B false-runs after A succeeds AND after A fails-but-mirror-says-nothing.
- **Design ref:** DESIGN §1 CloseReason rationale, §3 'Sync moves one field'; beads blocked_state.go:152; beads types.go:923-937 (IsFailureClose: no production callers); gascity dispatch/runtime.go:696-718, dispatch/retry.go:248-280
- **Fix:** Amend the design's sync contract (§3, plus §4 creation-time-already-closed): whenever the controller closes a mirror, copy the remote's outcome metadata trio — gc.outcome, gc.failure_class, gc.failure_reason (beadmeta.OutcomeMetadataKey / FailureClassMetadataKey / FailureReasonMetadataKey) — onto the mirror alongside status+CloseReason (e.g. via the existing updateMetadataAndClose pattern), and clear them on reopen. Reads stay 100% local; beadOutcomeFailed(mirror) then equals beadOutcomeFailed(remote) for every existing consumer (resolveBlockedOutcome/finalize, fanout, tally). Also correct the §1 CloseReason rationale to cite gascity's gc.outcome machinery as the real consumer instead of bd conditional-blocks semantics (which are reason-blind and IsFailureClose is dead code).

## [10] MEDIUM · PARTIAL · false-ready-classification

- **Design ref:** mirror-beads-DESIGN.md SS1 (ID row), SS2 creation guard; beads internal/storage/dolt/dependencies.go:15, internal/storage/domain/db/dependency.go:49

## [11] MEDIUM · PARTIAL · false-ready-classification
**The SS2 belt-and-braces 'assert via DepList that the edge did not land as external' is unimplementable on current surfaces — it passes in exactly the failure case it exists to catch, and SS6b's sweep can't detect external rows either**

(b) An external row with a genuinely foreign target is OMITTED from DepList entirely, so an existence-style assert can't even see what it's checking, and ensureBlockingDependency's dedup (control.go:284-295) re-adds it every controller pass. Corollary: the SS6b reconcile sweep must 'detect pre-existing blocking-type depends_on_external rows', but no gascity store surface and no bd CLI read surface exposes which column a dep row landed in — so SS7's claim that v2 'needs from bd: nothing new' is false.
- **Design ref:** mirror-beads-DESIGN.md SS2 (belt and braces), SS6b step 2, SS7 v2; beads internal/storage/issueops/dependencies.go:900-916; gascity internal/beads/bdstore.go:2496
- **Fix:** Amend mirror-beads-DESIGN.md: (1) SS2 — replace 'assert via DepList that the edge did not land as external' with a pre-DepAdd invariant check: fail the wiring op unless ExtractPrefix(dependentID) == ExtractPrefix(mirrorID) (bd's classifier, domain/db/dependency.go:41-52, is deterministic on the SOURCE bead's prefix vs target prefix — note it does NOT key on the store prefix, so also error when the dependent's own prefix differs from its store's Create prefix); optionally add a post-wire behavioral re-check via the dependent's is_blocked/Ready projection where the deployed bd emits it. (2) SS6b — specify the sweep over raw dependency records (batch `bd dep list <id1> <id2> --json` → types.Dependency via GetDependencyRecordsForIssues; gascity's existing DepListBatch parses this shape), classifying as external every blocking-type row whose target has an 'external:' prefix or a prefix different from its source's — and pin the batch raw-record JSON contract with a test, since bd silently falls back to the hydrated (row-dropping) shape when the raw path errors (cmd/bd/dep.go:806) or when given a single id.
